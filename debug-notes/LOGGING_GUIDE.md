## 日志与报错收集规范

这份文档主要解决两个问题：
- **出了问题时，快速知道去哪里看日志**
- **记录问题时，有统一的截图 / 文本格式，方便以后排查**

---

### 1. 常见日志来源

- **1）Next.js 终端日志**
  - 启动命令：`npm run dev`
  - 出问题优先看这里：
    - SyntaxError / TypeError
    - 哪个文件、哪一行报错
  - 建议：遇到红色报错，先整体截图保存到 `debug-notes/` 里，文件名包含日期。

- **2）浏览器控制台 (F12 -> Console)**
  - 适合排查：
    - React 运行时错误
    - 网络请求失败（接口 4xx / 5xx）
    - CSS / 布局相关的警告
  - 建议：
    - 报错时打开 Console，右键复制错误文本或截图，放到 `debug-notes/` 里。

- **3）网络面板 (F12 -> Network)**
  - 用来检查：
    - API 请求是否成功（状态码）
    - 返回数据结构是否符合预期
  - 调试步骤：
    1. 打开 Network 面板
    2. 触发页面操作（比如点击导航 / 加载某个 Section）
    3. 找到对应请求，看状态码 + Response

---

### 2. 建议的记录方式

每次遇到问题，可以在 `debug-notes/` 里新建一个 Markdown 记录，比如：

- 文件名示例：`2026-03-10-next-build-error.md`
- 内容结构建议：

```markdown
## 现象
- 描述你看到的内容 / 红色提示 / 页面表现

## 触发步骤
1. 打开页面 …
2. 点击 …

## 终端日志
```text
粘贴 `npm run dev` / `npm run build` 中的错误片段
```

## 浏览器控制台
- 粘贴关键错误或截图路径说明

## 自己的猜测（可选）
- 可能与哪个文件 / 哪次修改有关……
```

之后把这个文件发给 AI / 同事，可以极大降低沟通成本。

---

### 3. 平时开发的小习惯（推荐）

- **遇到红色错误，不要只截一半**：尽量包含文件路径 + 行号。
- **改动前后多用 Git / 备份**：方便对比「出问题前后的区别」。
- **一个问题一条记录**：不要把不同类型的问题混在一个文件里。
